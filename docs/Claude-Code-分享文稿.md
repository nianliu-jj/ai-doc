## 1. 开场引入（3 分钟）

### 1.1 故事：从「辅助工具」到「执行者」

**各位好，今天我想先讲一个真实的场景。**

我曾经问过一个团队：你们用 AI 编程工具做什么？

> "让它帮我补全代码。"
> "让它解释这段逻辑是什么意思。"
> "让它生成一些简单的函数。"

这些都是「写代码」层面的用法——AI 是一个高级查档员，一个智能剪贴板。

**但如果我告诉你，AI 可以直接帮你完成一个功能模块？**

```
过去：我想好了怎么做 → 告诉 AI → AI 生成代码 → 我审查修改
现在：告诉 AI 要什么 → AI 自己想办法 → 生成代码 → 运行验证 → 完成
```

这不是科幻。Claude Code 正在让这件事成为现实。

### 1.2 AI 编程工具的 2026 格局

经过两年多的发展，AI 编程工具已经形成了清晰的分工：

| 工具 | 定位 | 核心场景 |
|------|------|----------|
| **Claude Code** | 终端原生 AI 编程助手 | 项目级理解、复杂逻辑、终端工具链 |
| **Cursor** | IDE 内嵌 AI | 实时补全、Small-to-Medium 修改 |
| **GitHub Copilot** | IDE 插件 | 快速补全、简单函数生成 |
| **Claude.ai** | 网页对话 | 思考辅助、知识问答 |

**为什么 Claude Code 值得关注？**

Anthropic 官方对 Claude Code 的描述是：

> "Claude Code is low-level and non-prescriptive — it works with your existing workflow rather than forcing you into a particular approach."
>
> Claude Code 是低层且不强制特定工作流的——它与您现有的工作流程协同，而非强制您采用特定方式。

这句话的关键词是：**低层**和**不强制**。

- **低层**：直接操作文件系统、Git、终端，而不是躲在 IDE 后面
- **不强制**：你可以用它替换 VS Code，也可以完全不用 IDE

### 1.3 三个核心能力定义 Claude Code

```
┌─────────────────────────────────────────────────────────────┐
│                    Claude Code 三剑客                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   1. 终端原生                                                 │
│      └── 直接在命令行运行，与 Shell 环境深度集成                  │
│                                                              │
│   2. 上下文感知                                               │
│      └── 自动理解项目结构，读取相关文件，保持对话历史              │
│                                                              │
│   3. 工具链集成                                               │
│      └── 内置文件操作、Git 工作流、命令执行能力                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**一个类比：**

如果你把 AI 编程工具比作厨房：

- **Copilot** 像是一个帮你切菜的助手（你得自己决定做什么菜）
- **Cursor** 像是一个帮你颠勺的厨师（你得站在旁边指导）
- **Claude Code** 像是一个能把菜单变成整桌菜的团队（你只需要告诉要什么风格的菜）

## 2. 核心概念（7 分钟）

### 2.1 CLAUDE.md：AI 的「宪法」

如果要选一个功能来代表 Claude Code，我会选 **CLAUDE.md**。

这不是一个普通的配置文件，它是**给 AI 看的项目说明书**。

**CLAUDE.md 能做什么？**

```markdown
# 我的 Vue 3 项目

## 技术栈
- 前端: Vue 3 + TypeScript + Naive UI
- 构建: Vite
- 样式: Uno CSS

## 编码规范
- 组件命名: PascalCase（如 UserCard.vue）
- 样式类名: kebab-case
- 禁止使用: any 类型、Options API

## 工作流程
1. 新功能必须先写测试
2. 提交前运行 lint
3. PR 需要至少一人 review
```

**把这个文件放进项目后，Claude Code 会自动阅读并遵守。**

CLAUDE.md 有三层优先级：

```
1. ./.claude/CLAUDE.md    ← 项目级（最高）
2. ./CLAUDE.md             ← 根目录
3. ~/.claude/CLAUDE.md     ← 全局（最低）
```

**最佳实践：**

- 项目级配置放 `.claude/CLAUDE.md`，避免根目录混乱
- 全局配置记录通用偏好（如"总是用中文回复"）
- 项目配置覆盖全局配置

### 2.2 记忆系统：让 AI 记住「常识」

Claude Code 的记忆系统分为三层：

```
┌────────────────────────────────────────────────────────────┐
│                     记忆系统架构                              │
├────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐       │
│  │   短期记忆   │    │   工作记忆   │    │   长期记忆   │       │
│  │  (Session) │    │ (CLAUDE.md) │    │  (Memory/)  │       │
│  └─────────────┘    └─────────────┘    └─────────────┘       │
│        │                  │                  │              │
│        ▼                  ▼                  ▼              │
│   当前对话上下文        项目级知识        跨会话知识           │
│   只在当前会话有效     随项目携带         永久保存             │
│                                                             │
└────────────────────────────────────────────────────────────┘
```

**长期记忆的四种类型：**

| 类型 | 用途 | 示例 |
|------|------|------|
| `user` | 用户偏好 | "用户喜欢函数式编程" |
| `feedback` | 反馈记录 | "用户纠正过：不用 class" |
| `project` | 项目知识 | "这是微服务架构" |
| `reference` | 外部引用 | "Bug 跟踪在 Linear" |

**为什么这很重要？**

> "AI 的上下文像牛奶，越新鲜越好。"

这句话的意思是：随着对话进行，上下文会越来越长，有效信息密度越来越低。

记忆系统的价值是：**把常识从对话中抽离出来，让上下文只承载当前任务的相关信息。**

### 2.3 工具系统：AI 的「手」

Claude Code 内置了 6 大核心工具：

| 工具 | 功能 | 风险等级 |
|------|------|----------|
| **Read** | 读取文件内容 | 低 |
| **Write** | 写入文件内容 | 高 |
| **Edit** | 编辑文件 | 高 |
| **Bash** | 执行 Shell 命令 | 高 |
| **Grep** | 搜索文件内容 | 低 |
| **Glob** | 匹配文件路径 | 低 |

**风险等级决定是否需要用户确认：**

- 低风险工具（如 Read、Grep）：自动执行
- 高风险工具（如 Write、Edit、Bash）：需要用户确认

你可以在 `settings.json` 中调整权限策略：

```json
{
  "permissions": {
    "allow": ["Read", "Glob", "Grep", "Edit"],
    "deny": ["Bash"]
  }
}
```

### 2.4 会话与上下文管理

**上下文窗口**是 AI 每次对话能处理的 Token 上限。当上下文快满时，Claude Code 会自动压缩历史信息（保留关键决策，丢弃重复内容）。

**上下文优化的三个原则：**

1. **任务单一性**：一个会话专注一个任务
2. **文件精确性**：只给 AI 读需要的文件
3. **指令清晰性**：一次说清楚要什么

## 3. 实战技巧（10 分钟）

### 3.1 并行运行多个 Claude 实例

这是被严重低估的功能。

**场景：** 你在做一个大型重构，需要同时关注多个模块。

```bash
# 终端 1：重构用户模块
claude --cwd ./packages/user

# 终端 2：重构订单模块
claude --cwd ./packages/order

# 终端 3：同步更新类型定义
claude --cwd ./packages/shared-types
```

**进阶用法：**

```
终端窗口布局建议：
┌─────────────┬─────────────┐
│  Claude 1   │  Claude 2   │
│  (前端)      │  (后端)     │
├─────────────┼─────────────┤
│  Claude 3   │  Claude 4   │
│  (测试)      │  (文档)     │
└─────────────┴─────────────┘
```

> 小提示：5 个终端 + 5-10 个网页端 Claude 并行是完全可行的。

### 3.2 Plan 模式先行

当任务复杂时，先让 Claude 制定计划，再执行。

**操作：** `Shift + Tab` 两次进入 Plan 模式

```
用户: 重构整个认证模块

Claude [Plan 模式]:
我来制定一个详细的重构计划：

## 当前状态分析
- 现有认证逻辑在 auth.ts (500 行)
- 使用了过时的 JWT 库
- 没有 refresh token 机制

## 重构计划
1. [ ] 创建新的 auth/v2/ 目录
2. [ ] 引入新 JWT 库 (jsonwebtoken)
3. [ ] 实现 refresh token 机制
4. [ ] 迁移现有用户数据
5. [ ] 编写测试用例
6. [ ] 更新 API 文档
7. [ ] 删除旧代码

## 风险评估
- 数据迁移可能需要停机维护窗口
- 建议先在 staging 环境验证

是否按此计划执行？
```

**为什么 Plan 模式重要？**

- AI 的"思考过程"对你是可见的
- 你可以提前发现逻辑漏洞
- 避免做到一半推倒重来

### 3.3 团队协作：共享 CLAUDE.md

CLAUDE.md 可以纳入版本控制，实现团队标准化：

```bash
# 将 CLAUDE.md 加入 Git
git add CLAUDE.md .claude/
git commit -m "docs: 添加项目规范配置"

# 团队成员同步
git pull
```

**团队共享配置的典型内容：**

```markdown
# XX 项目团队规范

## 代码规范
- ESLint + Prettier 必须通过
- 提交信息遵循 Conventional Commits
- 禁止 console.log（用日志库代替）

## Git 工作流
- feature 分支从 main 创建
- PR 需要至少 2 人 review
- 合并前必须通过 CI

## 技术栈约束
- 禁止引入未审核的新依赖
- 数据库迁移需要 DBA review
```

### 3.4 自定义 Slash Commands

Claude Code 支持自定义斜杠命令，这是实现团队标准化的另一把利器。

**典型场景：** 一键完成"提交 → 推送 → 创建 PR"

```bash
# 在 ~/.claude/settings.json 中配置
{
  "aliases": {
    "/commit-push-pr": "git add -A && git commit -m '$ARG' && git push && gh pr create --title '$ARG' --body 'Auto-created by Claude'"
  }
}
```

**使用：**

```
用户: /commit-push-pr feat: 添加用户注册功能
→ 自动执行完整流程
```

### 3.6 Hook 系统：自动化工作流

Hook 允许你在特定事件发生时自动执行脚本。

**典型场景 1：自动格式化**

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write \"$CLAUDE_FILE_PATH\""
          }
        ]
      }
    ]
  }
}
```

**典型场景 2：敏感信息检查**

```bash
#!/bin/bash
# check-secrets.sh

FILE_PATH="$CLAUDE_FILE_PATH"

if grep -qE "(password|secret|api_key|token)" "$FILE_PATH"; then
  echo "⚠️ 警告: 文件可能包含敏感信息"
  exit 1
fi
exit 0
```

### 3.8 验证循环：质量提升 2-3 倍的关键

这是 **最重要的一条实战技巧**，也是 Boris Cherny（Claude Code 之父）反复强调的：

> **"Never trust code that hasn't been run."**
> 永远不要相信没有运行过的代码。

**验证循环的标准流程：**

```
1. 生成代码
      │
      ▼
2. 运行测试 ──失败──→ 修复错误 ──→ 运行测试
      │                         ↑
      │ 成功                    │
      ▼                         │
3. 运行 lint                       │
      │                         │
      ▼                         │
4. 手动验证功能                    │
      │                         │
      └──失败──→ 回到步骤 1 ─────┘
```

**为什么验证循环能提升 2-3 倍质量？**

因为 AI 生成的代码有相当比例会在以下环节暴露问题：

- 编译失败（语法错误）
- 测试失败（逻辑错误）
- 类型检查失败（类型不匹配）
- 运行时崩溃（边界条件）

**每提前一个环节发现问题，修复成本降低一个数量级。**

## 4. 多模型协作策略（5 分钟）

### 4.1 为什么需要多模型协作？

每个模型都有自己的擅长领域：

| 模型 | 核心优势 | 最佳场景 |
|------|----------|----------|
| **Claude (Sonnet/Opus)** | 深度推理、复杂逻辑、代码理解 | 核心算法、大型重构、架构设计 |
| **Gemini** | 多模态、设计稿理解 | UI 还原、前端代码生成 |
| **Codex/GPT** | 端到端自动化、Bug 修复 | 快速修复、简单任务 |

**类比：这不是让一个厨师做完整桌菜，而是让专业的人做专业的事。**

### 5.2 从「写代码」到「做项目」的思维转变

**开发者角色的演变：**

```
传统模式：
开发者 → 写代码（执行者）

AI 辅助模式：
开发者 → 写提示词（指导者）

Agent 模式：
开发者 → 编排任务（编排者）
           ↑
           │
      Claude Code 作为执行者
```

**这不是取代，而是升级。**

就像导航软件取代了纸质地图，但司机这个职业没有消失——只是工作内容从"记住路线"变成了"管理行程"。

### 5.3 学习路径建议

| 阶段 | 时间 | 目标 | 关键动作 |

## 附录

### A. 快捷键速查表

| 操作 | 快捷键 |
|------|--------|
| 进入 Plan 模式 | `Shift + Tab` × 2 |
| 继续上次会话 | `claude --continue` |
| 单次查询 | `claude -p "提示词"` |
| 列出所有会话 | `claude --list-sessions` |
| 查看配置 | `claude --config show` |

### B. 常用命令模板

```bash
# 在当前目录启动
claude

# 指定模型
claude --model claude-opus-4-6

# 单次查询并指定文件
claude -p "审查这段代码" src/auth.ts

# 从管道输入
git diff | claude -p "生成 commit message"

# 恢复会话
claude --continue
```
